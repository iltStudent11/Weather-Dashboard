# Rationale

The component hierarchy is intentionally route-first. `App.tsx` is the shell that defines navigation and route boundaries, while each page component owns one user task: `LocationsPage` for browsing/managing saved locations, `SearchLocationPage` for add/edit input and validation, `LocationDetailPage` for weather display, and `NotFoundPage` for fallback UX. I extracted these because they map directly to URL states and keep each file focused on a single responsibility. Cross-cutting logic is split out from UI: `weatherApi.ts` encapsulates API normalization and location parsing, and presentational concerns stay in the page components.

For state management, location list state now lives in a typed `LocationsContext` (`LocationsProvider` + `useLocationsContext`). I moved from lifted props to Context because location mutations are needed by multiple routed pages, and Context removes repeated prop threading through route elements. The provider exposes intent-based actions (`addLocation`, `editLocation`, `deleteLocation`) so consumers remain simple and declarative. Local state is still used where appropriate: form input/error state is local to `SearchLocationPage`, and fetch lifecycle state is local to the weather hook.

The custom hook `useWeather(query)` encapsulates asynchronous weather loading behavior. It centralizes the loading/success/error state machine, request error handling, and effect cleanup via cancellation guard to avoid stale state updates on unmount or query changes. This keeps `LocationDetailPage` mostly presentational while still supporting robust fetch behavior.

Routes are structured as `/` (list), `/search` (add/edit), `/location/:query` (detail), and `*` (not found). Data fetching is triggered in detail flow through `useWeather`, which calls `fetchWeatherByLocation` in the service layer. The result is a clean separation: routing controls what view is active, pages render state, and services/hooks handle network and transformation concerns.
