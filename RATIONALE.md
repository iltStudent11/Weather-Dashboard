# Rationale

The component hierarchy is centered around `App` as the route shell and state owner, with page-level components for each major user task: `LocationsPage` (list + row actions), `SearchLocationPage` (add/edit form), `LocationDetailPage` (weather display), and `NotFoundPage` (fallback route). I extracted these as separate components because each has a single clear responsibility and distinct UI lifecycle. `App` handles navigation and route composition, while pages focus on rendering and local interactions. Shared weather logic was moved out of components into `src/services/weatherApi.js` to keep UI code simple and to isolate API/parsing concerns.

For state management, I used lifting state to `App` rather than Context. The `locations` list is needed by a small, route-adjacent set of pages (`LocationsPage` and `SearchLocationPage`), and prop drilling depth is shallow (one level), so Context would add ceremony without much benefit. `App` exposes intent-based handlers (`onAddLocation`, `onEditLocation`, `onDeleteLocation`) so child pages remain declarative and side-effect light. Page-specific state stays local: form state/validation in `SearchLocationPage`, and fetch status/data/error in `LocationDetailPage`.

There are currently no custom hooks in the codebase. I kept logic in plain functions (`fetchWeatherByLocation`, location parsing helpers) because reuse is concentrated in the service layer rather than shared React lifecycle behavior. If fetch/status behavior expands across multiple pages, the next extraction would be a `useWeather(query)` hook encapsulating loading/success/error and cancellation.

Routes are structured by task: `/` for list management, `/search` for add/edit, `/location/:query` for detail, and `*` for not found. Data fetching is performed in `LocationDetailPage` through `fetchWeatherByLocation`, using a three-state pattern (`loading`, `success`, `error`) for explicit UX feedback. The service abstracts API differences (Open-Meteo geocoding/forecast and Zippopotam.us ZIP lookup) and normalizes returned location/weather data for consistent rendering.
